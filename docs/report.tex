\section{Objective}
% ---------------------------------------------------------------------

Build a wireless environmental monitoring system around the ESP32. A
battery-class sensor node reads its sensors, evaluates an on-board
environment-safety rule, and transmits over a 433~MHz LoRa link to a
second ESP32 acting as a gateway. The gateway acknowledges every packet
back over LoRa (a bidirectional link), then bridges the data over WiFi
to a cloud service on AWS, where a live web dashboard is reachable from
anywhere at \texttt{http://iot.gabr.online:8080/}.

This is project idea \#4 (IoT Environmental Monitor), deliberately
extended well past a single board so it exercises a broad range of
microprocessor concepts: a 1-wire digital sensor, the ADC, GPIO, SPI
driving an RF transceiver, a half-duplex bidirectional radio protocol,
the ESP32 WiFi stack with an HTTP client, a token-secured cloud ingest
API, and an explicit finite-state-machine firmware on each board with a
non-trivial evaluation/alarm branch.

The data path is one-directional for telemetry (Node~A $\rightarrow$
Gateway $\rightarrow$ cloud $\rightarrow$ browser), but the LoRa hop is
\emph{bidirectional}: the gateway transmits an \texttt{ACK,<seq>} back
to the node for every packet it receives, and the node confirms it. The
gateway joins a WiFi hotspot and HTTP-POSTs each packet (as JSON) to a
token-protected endpoint on an AWS EC2 instance, which serves the live
dashboard. The radio link is self-contained and keeps working (with
local USB output and LoRa ACKs) even if WiFi or the internet is down ---
the cloud uplink is an added stage, not a single point of failure.

\section{Hardware design}
% ---------------------------------------------------------------------

\subsection{Sensors and components}

\begin{longtable}{@{}llp{7.2cm}@{}}
\toprule
\textbf{Component} & \textbf{Interface} & \textbf{Role}\\
\midrule\endhead
ESP32-WROOM-32 DevKit ($\times$2) & ---       & MCU for the node + the gateway\\
LoRa Ra-01 (SX1278) ($\times$2)   & SPI       & 433~MHz transceiver (both boards)\\
DHT22 / AM2302 ($\times$1)        & 1-wire digital & Temperature + humidity (node A)\\
Photoresistor module ($\times$1)  & Analog (ADC) & Ambient light, day/night (node A)\\
\bottomrule
\end{longtable}

The photoresistor is a plain analog module (one \texttt{AO} pin to one
ADC channel) -- no library, no bus -- and it participates in the bonus
rule by distinguishing day from night. The gateway carries no sensors;
it is purely the LoRa$\leftrightarrow$WiFi bridge.

\subsection{Pin assignments}

The LoRa SPI bus is fixed by the ESP32's VSPI peripheral and is
identical on both boards:

\begin{center}\begin{tabular}{@{}llll@{}}
\toprule
SCK & MISO & MOSI & NSS / RST / DIO0\\
\midrule
GPIO18 & GPIO19 & GPIO23 & GPIO5 / GPIO27 / GPIO26\\
\bottomrule
\end{tabular}\end{center}

\begin{center}\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Board} & \textbf{Sensor} & \textbf{ESP32 pin}\\
\midrule
Node A & DHT22 DATA          & GPIO4 \\
Node A & Photoresistor AO    & GPIO34 (ADC1, input-only) \\
Gateway & (none)             & --- \\
\bottomrule
\end{tabular}\end{center}

\note{The Ra-01 is a 3.3~V part -- powered from the ESP32's 3V3 pin,
never 5~V, with a 100~nF ceramic across VCC--GND. GPIO34 is an
input-only ADC1 pin, the correct choice for an analog sensor and clear
of the LoRa SPI bus and the ESP32 strapping pins. A $\sim$17~cm wire
quarter-wave antenna on each Ra-01 ANT pad is required for a usable
link.}

\section{Software design}
% ---------------------------------------------------------------------

Both boards run a single explicit finite state machine in
\texttt{loop()}: an \texttt{enum} of states and a \texttt{switch}
executing one state per iteration. Each \texttt{case} maps 1:1 onto a
bubble in the diagrams below.

\subsection{Node A --- environmental safety + ACK (state machine)}

\texttt{EVAL} applies the bonus environment-safety rule; \texttt{ALARM}
is a genuine extra state (not an \texttt{if}) raised on enclosure
overheat \emph{or} a dark-and-humid (overnight condensation) condition.
After \texttt{SEND}, \texttt{WAITACK} flips the radio to receive and
listens $\sim$600~ms for the gateway's \texttt{ACK,<seq>}, printing
whether the round-trip was confirmed.

\subsection{Gateway --- LoRa$\leftrightarrow$WiFi bridge (state machine)}

On every received packet the gateway, in order: (1) transmits the LoRa
\texttt{ACK} \emph{first} (latency-critical, must land in the node's
$\sim$600~ms window), (2) prints the JSON to USB for local debugging,
(3) HTTP-POSTs the JSON to the cloud. WiFi is non-fatal: if it cannot
join the hotspot the gateway still LoRa-ACKs and prints locally, and
retries WiFi on the 5~s heartbeat.

\subsection{Cloud service}

A small Flask app (\texttt{dashboard/cloud\_app.py}) runs as an isolated
\texttt{systemd} service on an AWS EC2 instance, port~8080. It exposes
\texttt{POST /ingest} (gated by an \texttt{X-Token} header), folds each
posted JSON object into live state, and serves the same dashboard page
at \texttt{/} that the local serial dashboard uses. It is co-tenanted
on a box that already runs an unrelated service; isolation by port and
a dedicated service unit means the two never interfere.

The over-the-air payload is compact CSV
(\texttt{A,12,24.5,45.0,1830,0,0}); JSON is built once on the gateway.
Credentials are never hardcoded in committed code -- they live in a
gitignored \texttt{secrets.h} created from \texttt{secrets.example.h}.

\subsection{Key code segments}

\textbf{The bonus environment-safety rule} (Node A \texttt{EVAL}/\texttt{ALARM}):
\begin{lstlisting}
bool hot = (tempC >= TEMP_HOT_C);
bool condensation = (night == 1 && humPct >= HUM_HIGH_PCT);
state = (hot || condensation) ? S_ALARM : S_BUILD;
\end{lstlisting}

\textbf{The bidirectional ACK} -- gateway ACKs \emph{before} the slow POST:
\begin{lstlisting}
sendAck(seq);                 // LoRa TX back to the node (fast)
Serial.println(out);          // local debug
cloudPost(out);               // WiFi HTTP POST (may block up to 3 s)
\end{lstlisting}

\textbf{Node A confirming the ACK} (\texttt{WAITACK} state):
\begin{lstlisting}
if (strncmp(rx,"ACK,",4)==0 && strtoul(rx+4,0,10)==sentSeq) {
  ackOk = 1;                  // round-trip confirmed
}
\end{lstlisting}

\section{Cloud deployment (what we did)}
% ---------------------------------------------------------------------

The dashboard is hosted on an existing AWS EC2 instance (Amazon Linux,
us-east-1) that already runs an unrelated service on ports 80/443. To
avoid disturbing it, the IoT dashboard is fully isolated:

\begin{itemize}
\item A dedicated \texttt{systemd} service runs \texttt{cloud\_app.py}
in its own Python virtualenv on \textbf{port 8080} (the existing service
keeps 80/443 untouched).
\item One security-group rule opens TCP~8080; the ingest path is gated
by a shared \texttt{X-Token} secret (the ESP32 is behind a phone-hotspot
NAT, so source-IP restriction is not possible).
\item A Route~53 \texttt{A} record points \texttt{iot.gabr.online} at
the instance's elastic IP.
\item The WiFi password and the ingest token are kept out of version
control (gitignored \texttt{secrets.h}; the cloud reads the token from a
systemd environment variable).
\end{itemize}

\section{Additional features (bonus)}
% ---------------------------------------------------------------------

\begin{enumerate}
\item \textbf{Bidirectional LoRa link} --- the gateway ACKs every packet
and the node confirms it; the ACK is sent before the slow cloud POST so
the round-trip stays tight regardless of internet quality.
\item \textbf{WiFi cloud uplink + remote dashboard} --- the gateway
bridges LoRa to an AWS-hosted, token-secured dashboard reachable from
anywhere, not just a local serial monitor.
\item \textbf{Environment-safety alarm state} --- a real
\texttt{EVAL}/\texttt{ALARM} branch (overheat, or dark + high humidity),
with the photoresistor participating in the rule.
\item \textbf{Self-healing + graceful degradation} --- any sensor/radio
fault retries from \texttt{INIT}; WiFi loss degrades to LoRa-ACK + USB
without stopping the link.
\item \textbf{Hardware-independent demo path} --- \texttt{USE\_LORA 0}
streams straight to USB if a radio fails on the day; same dashboard.
\item \textbf{Secrets hygiene} --- no credentials in committed code;
\texttt{secrets.h} is gitignored with a committed example template.
\end{enumerate}

\section{Conclusion \& future work}
% ---------------------------------------------------------------------

The system meets the brief (idea \#4) and extends it into a
bidirectional wireless sensor link with an on-board safety state and a
cloud dashboard reachable from anywhere. The core engineering decisions
-- compact CSV on the radio with JSON built at the gateway, ACK before
cloud POST, WiFi as a non-fatal added stage, and secrets kept out of
version control -- made the firmware short, robust, and safe to publish.

Future work: encrypt the cloud uplink (HTTPS/TLS), add a gateway
$\rightarrow$ node command path on top of the existing ACK channel, and
add more nodes -- the protocol already namespaces by node ID, so an
extra node only needs a new ID and CSV schema.
